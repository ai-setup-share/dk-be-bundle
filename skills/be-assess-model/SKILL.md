# assess-model

model 레이어에서 요청된 도메인 타입을 처리할 수 있는지 평가한다.

## 입력

- 상위 assess 스킬들의 산출물 (model에 대한 요청을 통합)
- specs/models.md (기존 도메인 모델) — 프로젝트 루트 기준

## 평가 절차

### 1. 기존 model 확인

specs/models.md를 읽고, 요청된 도메인 타입이 있는지 확인한다.

- 완전 일치 → "충족". 작업 불필요.
- 기존 도메인에 필드/VO 추가로 처리 가능 → "부분 충족". 변경 계획 수립.
- 도메인 자체가 없음 → "미충족". 신규 생성 계획 수립.

### 2. 요청 항목별 분류

상위 레이어들의 요청을 항목별로 분류한다:

**A. 도메인 모델 (Write)**
- 새 도메인 필요? 기존 도메인 확장?
- 필드 구성, 팩토리 메서드, 상태 변경 메서드

**B. Read 모델**
- 새 Read 모델 필요? 기존 Read 모델 확장?
- Write 모델 필드 + JOIN 정보 (nickname 등)

**C. Identity VO**
- 새 도메인이면 Identity VO 필요

**D. enum**
- 새 enum 필요? 기존 enum에 값 추가?

**E. Value Object**
- 새 VO 필요? (Tag, LinkedContent 같은)
- 기존 공유 VO (Popularity 등) 변경 필요?

### 3. model 계획 수립

다음을 결정한다:

- 생성/변경할 클래스 목록
- 각 클래스의 필드 구성
- 팩토리 메서드 설계
- 상태 변경 메서드 설계 (새 인스턴스 반환)
- 기존 도메인 변경 시 영향 범위

## 규칙

- @Value immutable 객체로 정의한다.
- 상태 변경은 새 인스턴스를 반환한다. (with 패턴)
- 팩토리 메서드로 생성한다. (생성자 직접 호출 지양)
- 외부 의존성 없음. 순수 Java.
- soft delete 필드(isDeleted, deletedAt)는 모델에 포함하지 않는다. Entity 레이어에서만 관리.
- 기존 도메인 변경 시 as-is/to-be를 제시하고 유저 확인을 받는다.

## 산출물

### 1. 충족 여부

```
충족 / 부분 충족 / 미충족
```

### 2. 현재 레이어 작업 계획

**하위 레이어 없음. model이 최하위.**

```
- [신규] {도메인명}
  - 필드: {필드 목록}
  - 팩토리 메서드: {메서드 목록}
  - 상태 변경: {메서드 목록}

- [신규] {도메인명}Read
  - 필드: {Write 모델 필드 + JOIN 필드}

- [신규] {도메인명}Identity
  - 필드: Long {도메인명}Id

- [신규/변경] {enum명}
  - 값: {enum 값 목록}

- [신규] {VO명}
  - 필드: {필드 목록}
  - 팩토리 메서드: {메서드 목록}

- [변경] {기존 도메인명}
  - as-is: {기존 필드}
  - to-be: {변경 후 필드}
  - 영향: {이 변경이 상위 레이어에 미치는 영향}
```
